What is Ansible?

A configuration management tool for performing admin tasks â€” installing/removing packages, CRUD on users, managing services â€” across thousands of servers simultaneously.

Key Features

Idempotence (skips already-configured servers) Â· Parallel execution Â· Disaster recovery Â· Snowflake server support

Internet Gateway

Job Types Â· Freestyle vs Pipeline

Types of Jenkins Jobs

Freestyle Project

GUI-based click-through configuration. Good for quick experiments. Not version-controlled, hard to reuse. Avoid for real projects.

Pipeline â€” Scripted

Written in Groovy DSL. Uses node { } blocks. Imperative style â€” you control every step. Very flexible, but verbose and harder to read.

Pipeline â€” Declarative â­

Structured, YAML-like syntax. Uses pipeline { } block. Easier to read and maintain. Supports built-in post-conditions, parallel stages, agents. Recommended approach.

Multibranch Pipeline

Auto-creates a pipeline job for every branch containing a Jenkinsfile. Each branch gets its own build history. Perfect for feature-branch and PR workflows.

Organisation Folder

Scans an entire GitHub org or Bitbucket project. Auto-discovers all repos and creates Multibranch Pipelines for each. Used in large engineering teams.

Folder

Not a job type â€” a container for grouping related jobs. Supports folder-level credentials and views. Use to organise by team or environment.

Declarative Pipeline Â· Jenkinsfile

Declarative Pipeline â€” Structure & Syntax

Storing Secrets in Jenkins

Best Practices Checklist

Always use Jenkinsfile

Never configure pipelines via GUI for real projects. Jenkinsfile in Git = version control, code review, and full auditability of your CI/CD process.

Never hardcode secrets

Use Jenkins Credentials store or integrate AWS Secrets Manager / HashiCorp Vault. A leaked credential in Git history is extremely hard to fully remediate.

Tag images with BUILD_NUMBER

Never push :latest from CI. Use $BUILD_NUMBER or $GIT_COMMIT so every deployed image is traceable to an exact build and commit.

cleanWs() in post always

Always clean workspace after builds. Stale artifacts from previous builds cause mysterious failures and disk exhaustion on long-running agents.

Use parallel stages

Tests, security scans, and code quality checks can run simultaneously. Only run sequentially when one truly depends on another's output.

Keep stages small and named

One stage = one responsibility. Small, clearly named stages make it easy to see exactly which step failed in Blue Ocean UI without digging through a huge log.

GitHub Actions provides powerful CI/CD capabilities for Python Django projects. It enables automated testing, code quality checks, security scanning, and deployment directly from your GitHub repository.

GitHub Actions Fundamentals

Core Concepts
  • Workflow: Automated process triggered by events (push, PR, schedule)
  • Job: Set of steps that execute on the same runner
  • Step: Individual task within a job (run command, use action)
  • Runner: Server that executes workflows (GitHub-hosted or self-hosted)
  • Action: Reusable unit of code that performs a specific task

    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

  POETRY_VERSION: '1.6.1'

    runs-on: ubuntu-latest

        image: postgres:15
        env:
          POSTGRES_PASSWORD: postgres
          POSTGRES_DB: test_db
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432
      
      redis:
        image: redis:7-alpine
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 6379:6379
    
    steps:
    - name: Checkout code
      uses: actions/checkout@v4
      
    - name: Set up Python
      uses: actions/setup-python@v4
      with:
        python-version: ${{ env.PYTHON_VERSION }}

      with:
        version: ${{ env.POETRY_VERSION }}

        virtualenvs-in-project: true

        path: .venv

    - name: Install dependencies
      if: steps.cached-poetry-dependencies.outputs.cache-hit != 'true'
      run: poetry install --no-interaction --no-ansi
      
    - name: Run migrations
      run: |
        poetry run python manage.py migrate --settings=config.settings.test
      env:
        DATABASE_URL: postgres://postgres:postgres@localhost:5432/test_db

    - name: Run tests
      run: |

        REDIS_URL: redis://localhost:6379/0
        
    - name: Generate coverage report
      run: |
        poetry run coverage run --source='.' manage.py test --settings=config.settings.test
        poetry run coverage xml
      env:
        DATABASE_URL: postgres://postgres:postgres@localhost:5432/test_db

      uses: actions/setup-python@v4

      run: |
        python -m pip install --upgrade pip
        pip install black isort flake8 mypy bandit safety pylint
        
    - name: Format check with Black
      run: black --check --diff .
      
    - name: Import sorting check with isort
      run: isort --check-only --diff .
      
    - name: Lint with flake8
      run: |
        flake8 . --count --select=E9,F63,F7,F82 --show-source --statistics
        flake8 . --count --exit-zero --max-complexity=10 --max-line-length=88 --statistics
        
    - name: Type checking with mypy
      run: mypy . --ignore-missing-imports --check-untyped-defs
      
    - name: Security check with bandit
      run: bandit -r . -x tests/ --format json -o bandit-report.json
      continue-on-error: true
      
    - name: Upload bandit results
      uses: actions/upload-artifact@v3
      if: always()
      with:
        name: bandit-results
        path: bandit-report.json
        
    - name: Dependency vulnerability check
      run: safety check --json --output safety-report.json
      continue-on-error: true
      
    - name: Code complexity with pylint
      run: pylint --load-plugins=pylint_django --django-settings-module=config.settings.base $(find . -name "*.py" | grep -v migrations | head -20)
      continue-on-error: true

# .github/workflows/deploy.yml
name: Deploy to Production

on:
  push:
    tags: [ 'v*' ]

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment: staging
    
    steps:
    - name: Checkout code
      uses: actions/checkout@v4
      
    - name: Deploy to Heroku Staging
      uses: akhileshns/heroku-deploy@v3.12.12
      with:
        heroku_api_key: ${{ secrets.HEROKU_API_KEY }}
        heroku_app_name: "myapp-staging"
        heroku_email: "deploy@example.com"
        
    - name: Run staging tests
      run: |
        curl -f https://myapp-staging.herokuapp.com/health/ || exit 1

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production
    
    steps:
    - name: Deploy to AWS ECS
      run: |
        aws ecs update-service --cluster production --service django-app --force-new-deployment
        
    - name: Wait for deployment
      run: |
        aws ecs wait services-stable --cluster production --services django-app
        
    - name: Run smoke tests
      run: |
        curl -f https://api.example.com/health/ || exit 1
        python scripts/smoke_tests.py
      env:
        API_BASE_URL: https://api.example.com

  deploy-kubernetes:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production
    
    steps:
    - name: Set up kubectl
      uses: azure/k8s-set-context@v3
      with:
        method: kubeconfig
        kubeconfig: ${{ secrets.KUBE_CONFIG }}
        
    - name: Deploy to Kubernetes
      run: |
        sed -i 's|IMAGE_TAG|${{ github.sha }}|g' k8s/deployment.yaml
        kubectl apply -f k8s/
        kubectl rollout status deployment/django-app -n production
