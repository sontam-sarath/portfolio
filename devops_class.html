DevOps - Prashanth

DevOps is a cultural and technical approach that unifies software development (Dev) and IT operations (Ops) to enhance collaboration, streamline processes, and deliver software faster and more reliably. It focuses on breaking down silos between teams, enabling continuous integration, delivery, and deployment of software.

Key principles of DevOps include automation, collaboration, and continuous feedback. Automation reduces repetitive tasks like testing and deployment, ensuring efficiency and consistency. Collaboration fosters open communication between developers, operations teams, and stakeholders, aligning goals and improving problem-solving. Continuous feedback ensures that products evolve based on user needs and performance metrics.

DevOps practices rely on tools for source control (e.g., Git), CI/CD pipelines (e.g., Jenkins, GitLab CI), infrastructure as code (e.g., Terraform), and monitoring (e.g., Prometheus, Grafana). These tools enable rapid iterations, scalability, and resilience in the development lifecycle.

The benefits of DevOps include faster time to market, improved quality, and enhanced system reliability. Organizations adopting DevOps often experience better team morale and reduced downtime, as issues are detected and resolved proactively.

By integrating development and operations with a culture of shared responsibility and automation, DevOps accelerates innovation while maintaining stability and scalability.

Redirection and Piping
  • Redirection:
    • Standard input redirection: 0 <
    • Standard output redirection: 1 > >>
    • Error redirection: 2 > >>
    • Redirect Linux command output to a file
    • Example: date > date.txt, adds the date to the date.txt file
    • Example: echo "hi welcome" >> date.txt, appends a new line to date.txt
    • Error redirection example: data 2>> date.txt, captures errors
    • Use semicolons to add multiple commands in one line and redirect their output
    • Examples:
      • (date; data; ls; pwd) > date.txt: redirects standard output, skips errors
      • (date; data; ls; pwd) &> date.txt: redirects both standard output and errors
  • Piping:
    • Output of one command is input for another
    • Example: cat -n devops.txt | grep red, searches for "red" in devops.txt
    • Case-insensitive search: grep -i
    • Real-time example: docker ps -a | grep -i sarath > sarath_containers.txt
  • History:
    • Lists previously executed commands
    • Example: !10 runs command at line 10
  • File operations:
    • Last 5 lines: cat -n date.txt | tail -5
    • First 5 lines: cat -n date.txt | head -5
    • Lines 21-30: cat -n devops.txt | head -30 | tail -10

Cron Jobs and Shell Scripting
  • Cron Jobs
    : Schedule jobs at a particular time in Linux
    • Syntax to set cron jobs: minutes hours dayofthemonth month dayoftheweek /
    • For example: 30 20 10 12 2 - "At 20:30 on day-of-month 10 and on Tuesday in December."
    • Use crontab.guru for easy cron expression generation
    • To edit cron jobs: crontab -e
    • Examples:
      • Daily: 30 20 * * * - runs every day at 20:30
      • Every Tuesday: 30 20 * * 2 - runs every Tuesday at 20:30
      • 1st of every month: 30 20 1 * * - runs on the 1st of every month at 20:30
      • Every hour: 30 * * * * - runs every hour at minute 30
      • Every minute: * * * * * - runs every minute
  • Shell Scripting
    :
    • File extension: .sh or .bashrc
    • Start script with shebang: #!/bin/bash - defines the shell interpreter for the script
    • List available shell interpreters: cat /etc/shells
    • Document the script properly
    • Make the script executable: chmod 777 filename.sh
    • Execute the script: ./filename.sh
    • Enable debugging: set -x (shows command and output)
    • Disable debugging: set +x
    • Enable brace expansion: set -B (e.g., mkdir sss{1..232})
    • Disable brace expansion: set +B
    • Multiple comments in the script:

      • List installed packages: apt list --installed
      • Search for a file: sudo find / -name tomcat10
      • Uninstall a package: sudo apt remove tomcat10
      • Remove dependencies after uninstalling: sudo apt-get purge tomcat10
      • Remove unused packages: sudo apt autoremove

    • To completely uninstall a package and its dependencies:
      • sudo apt remove tomcat10
      • sudo apt-get purge tomcat10

      • sudo apt autoclean
  • Creating Users in Linux
    • Each user in Linux is assigned a User ID (UID):
      • UID 0-1000: System-generated users
      • UID > 1000: User-created users
    • Example user entry in /etc/passwd: 
      root:x:0:0:root:/root:/bin/bash
      • root: Username
      • x: Placeholder for password
      • 0: UID
      • 0: GID
      • root: Comment
      • /root: Home directory
      • /bin/bash: Shell type
    • List all users: cat /etc/passwd
    • Example to create a user: sudo useradd madhu (creates default user with UID 1001)
    • To specify user options during creation: 
      sudo useradd -s /bin/bash -m -d /home/prakash -c "developer" prakash
      • -s: Set shell type
      • -m: Create home directory
      • -d: Set home directory
      • -c: Add comment
    • Specify custom UID: sudo useradd -s /bin/bash -m -d /home/nanna -c "nanna" -u 9999 nanna
    • To check a specific user: cat /etc/passwd | grep -i nanna
    • To view all options available for useradd: man useradd
  • Shell Check

  • IP Address:
    • Sample: 23.34.45.56
    • IPv4 - 32 bits in size - 4 octets - each octet is 8 bits
    • The value of an IP address should range from (0-255).(0-255).(0-255).(0-255)
    • 23.306.55.1221 - Invalid IP address
    • 23.232.32.133 - Valid IP address
    • We can generate up to 232 IP addresses with above combinations - approx 4 billion IP addresses.
    • IPv6 - 128 bit - 8 octets - each octet 16 bit in size
    • Sample: aaa.bbb.ccc.ddd.eee.fff.ggg.hhh
    • (0-65536).(0-65536).(0-65536).(0-65536).(0-65536).(0-65536).(0-65536).(0-65536)
    • IPv6 can accommodate 2128 (340,282,366,920,938,463,463,374,607,431,768,211,456) number of IP addresses.
    • IPv6 ensures we will never run out of available IP addresses.
    • Current use cases: mobiles, IoT devices, car sensors, CCTV
  • IP Address Classes:
    • Class A: 0-126.0-255.0-255.0-255
    • Class B: 128-191.0-255.0-255.0-255

Ansible Notes
  • Ansible
    • Ansible is a configuration management tool.
    • Used to perform administration activities such as installing or removing packages, CURD operations on users, and managing services (start/stop/enable/delete).
    • Helps manage configuration on a large number of servers simultaneously.
    • For example, configuring 1000 servers manually would take considerable time, but Ansible allows this to be done efficiently.
    • Works using parallel execution, saving time and reducing errors.
  • Why Ansible?
    • Ideal for DevOps engineers managing large server infrastructures.
    • Can list and manage pre-requisites like Python Django, MySQL, Angular, PHP, etc.
    • Open source and owned by Red Hat (enterprise version: Ansible Tower).
    • Written in Python (initially Python 2, now transitioned to Python 3).
    • Default communication protocols: SSH for Linux and RDP for Windows.
  • Other Configuration Management Tools
    • Chef
    • Puppet
    • SaltStack
  • Push-Based vs Pull-Based Models
    • Ansible uses a push-based model, requiring no agent program on target servers.
    • Other tools (Chef, Puppet, SaltStack) use a pull-based model, requiring agents on all target servers to pull configurations from the main server.
  • Key Features of Configuration Management Tools
    • Provision of servers: Installing packages, CRUD operations on users, and managing services.
    • Handling snowflake servers: Supporting servers with varying configurations.
    • Idempotence: Avoids reconfiguring servers already in the desired state.
    • Disaster recovery: Quick recovery in case of issues on target servers.
  • Ansible Usage Approaches
    • Adhoc Commands:
      Use for simple, one-off commands on target servers.
    • Playbooks:
      Use YAML files to configure web servers and launch websites.
    • Roles:
      Use for large-scale operations requiring multiple tasks.

Steps to Set Up Ansible
  1. Install Ansible: 
                                                #!/bin/bash
                                            sudo apt update
                                            sudo apt install software-properties-common
                                            sudo add-apt-repository --yes --update ppa:ansible/ansible
                                            sudo apt install ansible
  2. Generate SSH identity for the server: 
    ssh-keygen -t rsa
    • By default, public/private RSA key pair is stored at: /home/ubuntu/.ssh/id_rsa
    • Identity of server is located at: /home/ubuntu/.ssh/id_rsa.pub
  3. Create other managed nodes (target nodes/servers).
  4. Add the public key from Ansible server (/home/ubuntu/.ssh/id_rsa.pub) to new nodes:
    • Append the key to /home/ubuntu/.ssh/authorized_keys on the managed nodes.
  5. Change the hostname of the Ansible controller: 
    sudo hostnamectl set-hostname ansible-controller && exec bash
  6. Open the hosts file on the Ansible controller and add the IP addresses of managed nodes: 
    sudo nano /etc/ansible/hosts
  7. Perform administrative activities on managed nodes using Ansible commands.
Ad-Hoc Commands in Ansible
  • Install default JDK on all managed nodes: 
    ansible all -m apt -a 'name=default-jdk state=present' -b
  • Add a user named "sarath" on all managed nodes: 
    ansible all -m user -a 'name=sarath state=present' -b
  • General command structure: 
    ansible all// -i inventoryfile -m module -a "command related to the module"

Terraform Overview
  • Terraform:
    An open-source Infrastructure as Code (IaC) tool developed by HashiCorp.
    • Uses a declarative configuration language called HashiCorp Configuration Language (HCL).
  • Key Features of Terraform:
    • Declarative Language:
      Describes the desired end state of infrastructure, and Terraform figures out how to achieve it.
    • Provider Support:
      Supports multiple cloud providers like AWS, Azure, Google Cloud, and on-premises solutions.
    • Execution Plans:
      Generates an execution plan showing changes before applying them.
    • State Management:
      Tracks the current state of infrastructure in a state file.
    • Modularity:
      Allows reusable modules for common infrastructure patterns.
  • Why Do We Need Terraform?
    • Automates infrastructure management to reduce manual effort and errors.
    • Manages multi-cloud and on-premises environments with a unified workflow.
    • Ensures consistency and versioning by storing configurations in version control systems.
    • Enables scalability by reusing configurations for complex environments.
    • Supports Infrastructure as Code (IaC) for better documentation, review, and testing.
    • Provides change management through execution plans.
    • Improves team collaboration by sharing configurations and state files.
    • Facilitates disaster recovery by defining infrastructure as code.
  • 5 Major Stages in Terraform:
    • init:
      Initializes Terraform to manage infrastructure.
    • validate:
      Checks syntax and validates the script.
    • plan:
      Verifies existing infrastructure and previews changes.
    • apply:
      Executes the Terraform script.
    • destroy:
      Releases or removes all resources mentioned in the script.
  • Terraform Installation:
    • Install Terraform on the server where Terraform activities will be performed.
    • Obtain AWS access and secret keys for managing infrastructure.
  • Useful Links:
    • terraform.io
    • AWS Terraform Provider
  • Commands to Install Terraform:
    • wget -O - https://apt.releases.hashicorp.com/gpg | sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg
    • echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
    • sudo apt update && sudo apt install terraform
  • Terraform Script Segments:
    • Provider Plugin Block:
      Adds information related to the cloud provider (e.g., AWS, Azure).
    • Provider Credential Block:
      Contains credentials for the target AWS account.
    • Resource Block:
      Specifies infrastructure to manage.
  • Example Terraform Script:
                                terraform {
                              required_providers {
                                aws = {
                                  source  = "hashicorp/aws"
                                  version = "~> 5.0"
                                }
                              }
                            }
                            
                            provider "aws" {
                              region     = "us-east-2"
                              access_key = "my-key"
                              secret_key = "my-key"
                            }

                              tags = {
                                Name = "ServerFromTerraform"
                              }
                            }
  • Steps to Run a Terraform Script:
    1. terraform init
    2. terraform validate
    3. terraform plan
    4. terraform apply (enter yes when prompted)
    5. terraform plan (re-run to compare infrastructure changes; should return no changes needed if nothing has been updated).
    6. terraform destroy (deletes resources).

    • Never publish your access keys.
    • Always use access keys through environment variables.
    • Steps to hide access keys:
      1. First Approach:
        Create a new file provider.tf and add access key information in it.
      2. Second Approach:
        Install AWS CLI in the source machine where Terraform is installed:
        1. sudo apt update
        2. sudo apt install curl -y
        3. curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
        4. sudo apt install unzip -y
        5. unzip awscliv2.zip
        6. sudo ./aws/install
        7. aws --version
        8. Add AWS CLI to path:
          • echo 'export PATH=$PATH:/usr/local/bin' >> ~/.bashrc
          • source ~/.bashrc
        9. Clean up: rm -rf awscliv2.zip aws
        10. Run aws configure and input the following:
          • AWS Access Key
          • AWS Secret Access Key
          • Region (optional)
          • Output Type (optional)
        11. Remove access key information from provider.tf and maintain the file with just region.
      3. Third Approach:
        Create environment variables:
        • export AWS_ACCESS_KEY_ID="my-access-key"
        • export AWS_SECRET_ACCESS_KEY="my-secret-access-key"
      4. Fourth Approach:
        Use Terraform Secret Manager.
      5. Fifth Approach:
        Use an interactive Terraform script that prompts the user to enter access keys during runtime.
    • Best Practices:
      • Always create access keys from user accounts instead of root accounts, as user accounts have limited permissions.
      • Add user access keys to CLI:
        • aws configure --profile=user_name
        • Enter access key and secret key.
      • Update provider.tf to use user access keys for Terraform tasks: 
        provider "aws" {
                                profile = "user_name"
                            }
  • Files Created by Terraform:
    • .terraform/ (Directory):
      • Created when you initialize a Terraform project using terraform init.
      • Contains essential files and plugins required for Terraform.
      • Includes:
        • Plugins for providers like AWS, Azure, etc.
        • Cached information about providers and module dependencies.
        • Cached modules if used.
      • Usage: Generally not modified manually, used internally during operations like terraform plan and terraform apply.
    • .terraform.lock.hcl:
      • Ensures consistent provider versions across machines or team members.
      • Automatically updated during terraform init or when provider versions change.
      • Usage: Commit this file to version control for consistency across environments.
    • terraform.tfstate:
      • Tracks the current state of your infrastructure.
      • Includes information about created resources, their attributes, and relationships.
      • Usage: Do not edit manually; manage changes via Terraform commands. Store in a remote backend for team collaboration and state locking.
    • terraform.tfstate.backup:
      • Automatic backup of the terraform.tfstate file.
      • Acts as a safety net for state recovery.
      • Usage: Typically not interacted with directly, used for manual recovery if needed.

Virtual Private Cloud (VPC) in AWS
  • Core Features of VPC:
    • Subnetting: Divide the VPC into public and private subnets.
    • IP Addressing: Define IP address ranges using CIDR (e.g., 10.0.0.0/16).
    • Route Tables: Control network traffic within the VPC.
    • Internet Gateway: Enables communication between VPC resources and the internet.
    • NAT Gateway/Instance: Allows private subnet instances to access the internet securely.
    • Security Groups and Network ACLs: Control traffic at the instance and subnet levels.
  • Use Cases:
    • Host scalable applications.
    • Enable hybrid cloud configurations with VPN or AWS Direct Connect.
    • Implement multi-tier architectures with separate subnets.
    • Secure networking with private subnets and tight access control.
  • VPC Components:
    • VPC Peering: Connect two VPCs privately.
    • Transit Gateway: Hub for connecting multiple VPCs and networks.
    • Elastic IP (EIP): Static public IP address for resources.
    • AWS PrivateLink: Secure connectivity between VPCs and services.
    • Endpoints: Private access to AWS services like S3 and DynamoDB.
  • Advantages:
    • Customization of network configurations.
    • Scalability to accommodate workloads.
    • Isolation for resources in a dedicated network.
    • Robust security controls.
  • Example Scenario:
    • Public-facing web servers in a public subnet with an internet gateway.
    • Databases in a private subnet for isolation.
    • NAT gateway for secure internet access by private resources.

Maven
Overview
  • Written in Java
  • A build tool developed by Apache Software Foundation
  • Open source
  • Used to compile and build Java-based projects
  • Uses pom.xml for instructions
    • Project Object Model Extensible Markup Language
    • Contains all build activities:
      • Dependencies (e.g., compiler, test plugins)
      • How to package: .jar, .war, .ear
  • Alternatives: build.gradle, setup.py, makefile (for non-Java projects)
How Maven Works
  • Maven build lifecycle comprises three stages:
    • Default
    • Site
    • Clean
  • Default lifecycle phases:
    • Validate: mvn validate - Checks files in src/, test/, and pom.xml.
    • Compile: Compiles Java files into .class files.
    • Test: Runs unit tests in test/.
    • Package: Compiles data into a single product (.war, .jar, .ear).
    • Verify: Ensures packaged content is synced.
    • Install: Installs the package locally.
    • Deploy: Deploys artifacts into a private registry.
  • Site: mvn site - Generates documentation as a web page (not widely used).
  • Clean: mvn clean - Clears earlier versions when a new version is launched.
Packages Created by Maven
  • .jar: (Java Archive) Packages all .class files.
  • .war: (Web Archive) Used for Java-based web applications; a combination of .jar files and web content.
  • .ear: (Enterprise Archive) Used for high-end applications; a combination of .jar and .war files.
Repositories in Maven
  • Local: Maintains in local machine, shared with other developers.
  • Remote: Packages in build repositories.
  • Central: Maintained by the Maven community.
  • During the build process, all required dependencies are downloaded from repositories mentioned in pom.xml.

For more information, visit the Maven Lifecycle Guide.
